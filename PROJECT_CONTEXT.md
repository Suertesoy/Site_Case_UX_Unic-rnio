# PROJECT_CONTEXT.md

## Contexto geral

Este projeto é o protótipo navegável do APP UNIEDU, uma solução mobile gamificada para apoiar a jornada de aprendizagem de alunos em cursos online.

O produto simula uma experiência de aplicativo com foco em:

aprendizado contínuo  
jornada de estudos  
gamificação  
pontos e recompensas  
loja interna  
ranking  
comunidade  
aulas e materiais de apoio  
anotações  
modo claro e modo escuro  

O objetivo atual é tirar o projeto da fase de protótipo e prepará-lo para deploy na Vercel, mantendo a experiência visual atual o mais fiel possível.

## Estado atual do projeto

O projeto veio de um protótipo gerado/evoluído a partir do Figma Make.

A estrutura principal usa React, Vite, TypeScript e React Router.

O app possui contextos separados para responsabilidades como loja, nível, aulas, notas e tema.

O protótipo é apresentado em um frame mobile, com aparência próxima de um iPhone 16.

## Direção de produto

A experiência deve parecer divertida, acolhedora, motivadora e educacional.

A gamificação não deve parecer infantil demais. Ela precisa transmitir progresso, conquista e recompensa, mas ainda com uma sensação de produto bem desenhado.

O foco é demonstrar uma solução de UX/UI para portfólio, então a qualidade visual, consistência de fluxo e clareza da experiência são mais importantes do que criar funcionalidades complexas neste momento.

## Prioridades atuais

Preparar o projeto para deploy na Vercel.

Garantir que npm install e npm run build funcionem.

Evitar quebrar rotas, providers, contextos e estilos globais.

Manter o visual atual, exceto quando uma alteração visual for solicitada explicitamente.

Corrigir bugs técnicos sem redesenhar o produto sem autorização.

Melhorar organização do código aos poucos, sem grandes refatorações arriscadas.

## Cuidados importantes

Não alterar a identidade visual sem pedir aprovação.

Não modificar o dark mode global sem explicar impacto.

Não mover arquivos ou renomear componentes sem justificar.

Não transformar o protótipo em outro produto.

Não adicionar dependências novas sem necessidade clara.

Não alterar rotas existentes sem validar impacto.

Não apagar dados mockados sem criar substituição equivalente.

## Expectativa de trabalho

Antes de qualquer alteração relevante, apresente um plano com:

arquivos que serão alterados  
motivo da alteração  
risco da mudança  
impacto visual esperado  
como testar depois  

Só implemente depois de aprovação.
